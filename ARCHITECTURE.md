# ARCHITECTURE.md — ProfessorGPT System Design

## System Overview

ProfessorGPT follows a five-stage processing pipeline. Each stage is independently modular, making it easy to upgrade components (e.g., swap TF-IDF for a vector database) without rewriting the whole system.

```
┌─────────────┐    ┌──────────────┐    ┌───────────────┐    ┌─────────────────┐    ┌──────────────┐
│  Raw Input  │───▶│ Preprocessing│───▶│ Semantic Index│───▶│ LLM Generation  │───▶│ UI / Output  │
│  .txt / .mp3│    │ Clean & chunk│    │ TF-IDF vectors│    │ Claude Opus API │    │ Streamlit    │
└─────────────┘    └──────────────┘    └───────────────┘    └─────────────────┘    └──────────────┘
```

## Component Breakdown

### 1. Input Layer (`transcript_processor.py`)

- Accepts `.txt` transcripts pasted or uploaded directly
- Strips caption artifacts (`[Music]`, timestamps, `(applause)`)
- Normalizes Unicode, collapses whitespace, trims lines
- **Optional:** Integrates with OpenAI Whisper API to transcribe `.mp3`/`.mp4` files into text before processing

### 2. RAG Engine (`rag_engine.py`)

The retrieval-augmented generation engine ensures chat answers are grounded in actual lecture content — not hallucinated.

**Steps:**
1. **Chunking** — transcript split into 300-word windows with 50-word overlap
2. **TF-IDF Vectorization** — each chunk converted to a sparse vector
3. **Cosine Similarity Retrieval** — query vectorized; top-3 most relevant chunks returned
4. **Context Injection** — retrieved chunks injected into Claude's system prompt

```
Query ──▶ Tokenize ──▶ TF-IDF Vector ──▶ Cosine Similarity ──▶ Top-K Chunks ──▶ Claude Prompt
```

**Production upgrade path:** Replace `rag_engine.py` with `sentence-transformers` for dense embeddings + ChromaDB or Pinecone for vector storage. The interface (`retrieve(query)`) stays the same.

### 3. Content Generation (`app.py` — generation functions)

Each study tool is generated by a dedicated LLM call with a carefully engineered prompt:

| Feature | Prompt Strategy | Output Format |
|---------|----------------|---------------|
| Smart Notes | Section-by-section decomposition | Markdown |
| Flashcards | Term extraction + definition synthesis | JSON array |
| Q&A Pairs | Question generation + grounded answers | JSON array |
| Exam Questions | Distractor generation + explanation | JSON array |
| RAG Chat | Grounded Q&A with transcript context | Conversational |

All prompts include:
- **Role priming** — "You are an expert academic note-taker"
- **Output constraints** — exact format required (JSON, Markdown)
- **Grounding instruction** — base outputs on provided transcript
- **Hallucination mitigation** — explicit instructions to not add unsupported information

### 4. UI Layer (`app.py` — Streamlit)

- Tab-based navigation for each feature
- Session state preserves all generated content across tab switches
- Regenerate buttons allow one-click refresh of any feature
- Download buttons export notes as `.md`

  ├─ _chunk()      → list of 300-word overlapping segments

        │

|-------|-----------|-----------|
| Backend language | Python 3.11 | Rich AI/ML ecosystem |
| Web framework | Streamlit | Rapid prototyping, no frontend code needed |
| LLM API | Anthropic Claude Opus | State-of-the-art instruction following |

| Dev environment | Cursor (AI-assisted IDE) | Accelerates development |

---

## Scalability Considerations

- **Stateless design** — all state in `st.session_state`; easy to move to Redis for multi-user
- **Modular RAG** — `rag_engine.py` has a clean interface (`retrieve(query)`) — swap internals without touching app logic
- **Prompt templates** — centralized in generation functions, easily modified for different course types
- **Cost optimization** — embeddings/chunks cached; model calls minimized by caching results in session state

---

## Security

- API key loaded from environment variable (`.env`) — never hardcoded
- Uploaded files processed in memory — not persisted to disk
- No user data stored server-side in the current implementation
